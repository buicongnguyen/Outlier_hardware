1. DSP & Signal Processing

Turning continuous signals into numbers and back without lying about them: sampling, quantization, filtering, and frequency analysis. Used everywhere a sensor, microphone, or radio meets software — audio processing, communications, and sensor filtering.

Key terms

Sampling
Measuring a continuous signal at regular intervals (the sample rate fs), turning it into a discrete sequence x[n].
Nyquist rate / frequency
To represent a signal with maximum frequency fmax you must sample at fs > 2·fmax. The Nyquist frequency fs/2 is the highest frequency the sampled system can represent.
Aliasing
When content above fs/2 is sampled, it folds back and masquerades as a lower frequency. Once aliased, it is indistinguishable from a real low-frequency component — unrecoverable.
Anti-aliasing filter
An analog lowpass filter placed before the ADC to remove content above fs/2. It must be analog: digital filtering after sampling is too late.
Quantization
Rounding each sample to one of 2N levels for an N-bit converter. The rounding error behaves like noise; each extra bit adds ≈6.02 dB of SNR (≈6.02·N + 1.76 dB for a full-scale sine).
Convolution
y[n] = Σ h[k]·x[n−k]: sliding the (flipped) impulse response over the input. Any LTI system's output is its input convolved with its impulse response. Convolution in time equals multiplication in frequency.
Impulse response h[n]
A system's output when fed a single unit impulse. It fully characterizes an LTI system.
FIR filter
Finite Impulse Response: output depends only on current and past inputs (no feedback). Always stable, and can have exactly linear phase if the coefficients are symmetric.
IIR filter
Infinite Impulse Response: uses feedback from past outputs. Far fewer coefficients for a sharp response, but can go unstable and has nonlinear phase.
DFT / FFT
The Discrete Fourier Transform maps N time samples to N frequency bins. The FFT is a fast algorithm for the same result: O(N log N) instead of O(N²).
Frequency bin
One output of an N-point DFT. Bin spacing (frequency resolution) is fs/N — longer captures resolve finer frequency detail.
Windowing
Multiplying a signal block by a taper (Hann, Hamming, Blackman) that goes to zero at the edges before taking the FFT, to reduce spectral leakage.
Spectral leakage
When a block does not contain a whole number of periods, the implied edge discontinuity smears a tone's energy across many bins.
Z-transform
The discrete-time analogue of the Laplace transform. z−1 is a one-sample delay; a filter's transfer function H(z) is a ratio of polynomials whose roots are its zeros and poles.
Poles and zeros
Roots of the transfer function's denominator and numerator. A causal filter is stable only if every pole lies strictly inside the unit circle.
Fixed-point arithmetic
Integers with an implied scale (e.g. Q15: 16-bit values representing [−1, 1−2−15]). Cheap and deterministic, but you manage range, overflow, and precision yourself.
Floating-point arithmetic
Sign/exponent/mantissa format with huge dynamic range and automatic scaling, at higher hardware cost. Standard on DSPs/CPUs with FPUs; fixed-point still rules cheap MCUs and dedicated hardware.
Decimation / interpolation
Reducing or raising the sample rate. Decimation by M requires lowpass filtering first, or content above the new Nyquist limit aliases.

Core concepts

Sampling and the Nyquist limit

Sampling is only lossless if nothing above fs/2 reaches the ADC. That is why every real acquisition chain starts with an analog anti-aliasing filter, and why audio CDs use 44.1 kHz for a ~20 kHz bandwidth. A component at frequency f > fs/2 appears at the alias frequency |f − k·fs| — e.g. 7 kHz sampled at 10 kHz shows up as 3 kHz. Pitfall: no amount of post-processing can undo aliasing; the information is gone.

Quantization and bit depth

An N-bit ADC gives roughly 6.02·N + 1.76 dB SNR for a full-scale sine (16-bit ≈ 98 dB, 12-bit ≈ 74 dB). Headroom is a tradeoff: signals near full scale maximize SNR but risk clipping, which creates harsh harmonic distortion — far worse than quantization noise. Dither (tiny added noise) decorrelates quantization error from the signal at low levels.

FIR vs IIR tradeoffs

FIR: no feedback, unconditionally stable, exactly linear phase with symmetric taps (all frequencies delayed equally — critical for data/comm pulses and audio crossovers), but sharp filters need many taps. IIR: feedback gives sharp responses with few coefficients (a biquad has 5), but phase is nonlinear and poles can drift outside the unit circle when coefficients are quantized. High-order IIR filters are built as cascaded biquads precisely to limit that sensitivity.

FFT, resolution, and windowing

An N-point FFT at sample rate fs gives bins fs/N apart — resolution costs observation time. A tone that does not land exactly on a bin leaks into neighbors; windows trade a wider main lobe (worse resolution) for much lower sidelobes (less leakage). Rectangular = best resolution, worst leakage; Hann/Hamming are the everyday compromise; Blackman when you need to see a weak tone next to a strong one.

Z-transform and stability

Design and analysis happen in the z-domain: H(z) = B(z)/A(z), where zeros shape notches and poles shape resonances. Stability test: all poles inside the unit circle. Poles close to the circle mean high gain, long ringing, and high sensitivity to coefficient rounding — the classic way a filter that worked in double precision blows up in 16-bit fixed point.

Fixed-point in practice

Q-format numbers (Q15, Q31) make multiplies cheap but every sum can overflow. Rules of thumb: keep track of worst-case gain through the pipeline, use guard bits in accumulators (e.g. 40-bit MACs), and saturate instead of wrapping — a wrapped overflow jumps from +max to −max, which in audio is a loud click and in a control loop can command a violent actuator reversal.

2. GPU Kernels

Writing code for thousands of threads at once: the SIMT execution model, the memory hierarchy, and the throughput mindset. Vocabulary below is CUDA's; OpenCL/HIP concepts map one-to-one (warp ≈ wavefront, shared memory ≈ local memory, block ≈ work-group).

Key terms

Kernel
A function launched from the host that runs on the device, executed by every thread in the launch grid.
SIMT
Single Instruction, Multiple Threads: groups of threads execute the same instruction in lockstep on different data. Each thread has its own registers and can branch — at a cost.
Thread
The smallest unit of execution. Identified inside a kernel by its thread and block indices, from which it computes which data element it owns.
Warp
A fixed group of 32 threads scheduled together. The hardware issues instructions per warp, not per thread.
Thread block
A group of up to 1024 threads that runs on one SM, shares shared memory, and can synchronize with a barrier. Blocks must be independent of each other.
Grid
All blocks of one kernel launch. There is no grid-wide barrier inside a kernel — cross-block agreement requires a new kernel launch (or atomics/cooperative groups).
Global memory
Device DRAM, visible to all threads. High bandwidth but high latency (~400–800 cycles); the usual performance bottleneck.
Shared memory
Small (tens of KB) on-chip scratchpad, private to a block, roughly 20–30× lower latency than global. Explicitly managed by the programmer for data reuse.
Registers
Fastest storage, private per thread. A finite per-SM pool — heavy register use limits how many threads can be resident.
Constant memory
Small cached read-only region, fastest when every thread in a warp reads the same address (a broadcast).
Coalesced access
When the 32 threads of a warp touch consecutive addresses, the hardware merges them into a few wide transactions. Strided or scattered access multiplies memory traffic.
Bank conflict
Shared memory is split into 32 banks. Two threads of a warp hitting different addresses in the same bank serialize; a common fix is padding an array by one element.
Occupancy
Ratio of resident warps on an SM to the hardware maximum, limited by registers, shared memory, and block-slot usage. Enough occupancy lets the scheduler hide memory latency.
Warp divergence
When threads of one warp take different branch paths, the warp executes each path serially with the other threads masked off.
Barrier (__syncthreads)
Block-level synchronization point: every thread of the block must reach it before any proceeds. Calling it inside divergent branches deadlocks or corrupts results.
Atomic operation
Read-modify-write (e.g. atomicAdd) that is safe under concurrency. Correct but serializing when many threads target the same address.
Reduction
Combining N values into one (sum, max). Done as a tree: each step halves the active threads, giving log₂(N) steps instead of N.
Host ↔ device transfer
Copies over PCIe (~tens of GB/s) — often slower than the kernel itself. Batch transfers, keep data resident on the GPU, use pinned memory and streams to overlap copy with compute.
Roofline model / arithmetic intensity
Plot of achievable FLOP/s vs arithmetic intensity (FLOPs per byte moved). Below the ridge point a kernel is memory-bound; above, compute-bound. Tells you which optimization can possibly help.

Core concepts

Think in warps, not threads

The scheduler issues instructions to 32-thread warps. An if that splits a warp runs both sides serially (divergence); a warp asking for 32 scattered addresses pays for 32 transactions. Fast kernels arrange work so threads of a warp branch together and read neighboring data. Divergence between different warps costs nothing — only within a warp.

Memory hierarchy is the game

Registers (per thread) → shared memory (per block) → L2 → global DRAM, each level roughly an order of magnitude slower. Most kernels are memory-bound, so the core skill is data reuse: stage a tile of global memory into shared memory once, synchronize, then let the whole block reuse it — the classic tiled matrix-multiply pattern. Pitfall: forgetting the second __syncthreads() before overwriting the tile.

Coalescing and layout

Warp-neighbor threads should access memory-neighbor addresses. This is why GPU code prefers structure-of-arrays over array-of-structures: with AoS, thread i reading field .x strides over the other fields and wastes most of each 128-byte transaction. A transposed or column-major access pattern can cut effective bandwidth by an order of magnitude.

Occupancy and latency hiding

A global load stalls a warp for hundreds of cycles; the SM hides that by switching to other resident warps. If registers or shared memory per block are too high, too few warps fit and the SM idles. But occupancy is a means, not a score — past the point where latency is hidden, more warps can even hurt by shrinking cache per thread. Measure, don't maximize.

Synchronization has scopes

__syncthreads() only synchronizes one block. Blocks may run in any order, on different SMs, and possibly not concurrently — so "block 0 writes, block 1 reads" is a race. Grid-wide ordering comes from ending the kernel and launching another (kernel launches on one stream are ordered), or carefully with atomics. Reductions therefore finish per-block, then combine partial results in a second kernel or with atomicAdd.

Roofline thinking

Before optimizing, compute arithmetic intensity = FLOPs / bytes moved. SAXPY (y = a·x + y) does 2 FLOPs per 12 bytes — hopelessly memory-bound, so tuning math throughput is pointless; only moving fewer bytes helps. Dense matrix multiply reuses each element O(N) times — compute-bound once tiled. The roofline tells you which wall you are actually hitting, and PCIe transfers sit below both: keep data on the device across kernels.

CUDA in practice — running real work (FFT example)

Anatomy of a CUDA program

Every CUDA program follows the same five-step rhythm: allocate device memory, copy inputs up, launch the kernel with a grid/block shape that covers the data, copy results back, free. Two things bite beginners: kernel launches are asynchronous (the CPU races ahead — a following cudaMemcpy or explicit sync is what waits), and the grid is rounded up, so the kernel needs an if (i < n) guard for the tail threads.

Libraries before hand-written kernels

For standard building blocks, vendor libraries are faster than anything you will write: cuFFT (FFTs), cuBLAS (matrix math), cuDNN (convolutions), Thrust/CUB (sort, scan, reduce). They are tuned per GPU generation. The professional pattern is: compose library calls for the heavy math, and write small custom kernels only for the glue — the pointwise steps between library stages.

Plan once, execute many

cuFFT (like FFTW) separates expensive setup from cheap execution: cufftPlan1d() chooses algorithms and allocates workspace once; cufftExec*() then runs at full speed as often as you like. Creating a plan per call is a classic performance bug. For many small FFTs, use a batched plan (cufftPlanMany) — one launch processing 1000 signals beats 1000 launches by a wide margin.

Keep the pipeline on the device

A GPU FFT that copies data host→device, transforms, and copies back is usually slower than the CPU — PCIe eats the win. The payoff comes from chaining: copy the signal up once, run forward FFT → filter → inverse FFT → analysis entirely on device buffers, and copy only the final result down. Design rule: move the pipeline to the GPU, not one function.

A complete CUDA workflow — SAXPY, the "hello world" kernel

__global__ void saxpy(int n, float a, const float *x, float *y) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;   // this thread's element
    if (i < n) y[i] = a * x[i] + y[i];               // guard the tail!
}

// host side — the 5-step rhythm of almost every CUDA program:
cudaMalloc(&d_x, bytes);                             // 1. allocate on device
cudaMemcpy(d_x, h_x, bytes, cudaMemcpyHostToDevice); // 2. copy inputs up
int block = 256, grid = (n + block - 1) / block;     // 3. cover n with threads
saxpy<<<grid, block>>>(n, 2.0f, d_x, d_y);           // 4. launch — asynchronous!
cudaMemcpy(h_y, d_y, bytes, cudaMemcpyDeviceToHost); // 5. copy back (this waits)

Finding the bottleneck — and designing it away

Profile the timeline first

Before touching any kernel, run Nsight Systems (nsys profile ./app) and look at the timeline: how much wall time is memcpy, how much is kernels, and how much is gaps where the GPU sits idle waiting for the CPU? Optimizing a kernel that is 10% of the timeline caps your win at 10% — the timeline tells you where the time actually goes, which is frequently not where you guessed.

Then deep-dive one kernel

Nsight Compute (ncu) profiles a single kernel: memory throughput vs peak, compute throughput vs peak, occupancy, and warp-stall reasons. The first-look rule: memory pipe near peak → memory-bound; compute pipe near peak → compute-bound; neither near peak → latency-bound (not enough parallel work in flight — the subtle case).

Back-of-envelope beats guessing

Effective bandwidth = bytes read + written ÷ kernel time. A kernel moving 800 MB in 10 ms achieves 80 GB/s; on a GPU with 800 GB/s peak, that screams latency/access-pattern problems, while 700 GB/s means the kernel is already near the roof and only algorithmic changes (move fewer bytes) can help. Two lines of arithmetic tell you which fight to pick.

Amdahl's law rules the pipeline

If 20% of the wall time is a serial CPU stage, the best possible overall speedup is 5× no matter how infinite the GPU is. Bottleneck removal is iterative: fix the biggest one and the second-biggest becomes the new limit. This is why end-to-end timeline profiling is repeated after every change — the bottleneck moves.

3. Embedded & Firmware

Software that talks to hardware directly: registers, interrupts, DMA, serial buses, and the bare-metal-vs-RTOS decision — plus the cross-compile → flash → debug loop that everything else depends on.

Key terms

Microcontroller (MCU)
A single chip with CPU core, flash, SRAM, and peripherals (e.g. ARM Cortex-M). MHz-class clocks, KB–MB memory, usually no MMU; runs bare metal or an RTOS with deterministic timing.
Application processor
GHz-class chip with MMU and external DRAM (e.g. Cortex-A) running a full OS such as Linux. Vastly more capable, but boot time, power draw, and scheduling jitter make it a poor fit for hard real-time control.
Memory-mapped register
A peripheral control/status word that appears at a fixed address; ordinary load/store instructions configure the hardware. Datasheet + address = driver.
volatile
C qualifier telling the compiler a value can change outside normal program flow (hardware register, ISR-shared variable), so every access must really happen — no caching in registers, no reordering away.
Interrupt / ISR
A hardware event that suspends the current code and jumps to an Interrupt Service Routine. The controller (e.g. NVIC) handles prioritization and nesting.
Interrupt latency
Time from the hardware event to the first ISR instruction — grows with critical sections that disable interrupts and with higher-priority ISRs running.
DMA
Direct Memory Access: a peripheral engine that moves data between memory and peripherals without CPU involvement, raising one interrupt per buffer instead of per byte.
GPIO
General-purpose I/O pin, configurable as input or output, often with pull-ups/downs and pin-change interrupts.
Timer / PWM
Hardware counters driven by a prescaled clock. PWM toggles a pin with a set period and duty cycle — the standard way to drive LED brightness, motor speed, and servos with zero CPU load.
UART
Asynchronous serial: no shared clock; both sides agree on a baud rate and resynchronize on each start bit. Two wires (TX/RX), point-to-point.
SPI
Synchronous full-duplex master/slave bus: SCLK, MOSI, MISO, plus one chip-select per slave. Simple and fast (tens of MHz), at the cost of pins.
I2C
Two-wire (SDA/SCL) addressed multi-drop bus with open-drain lines and pull-up resistors, ACK per byte, ~100–400 kHz typical. Many devices on two pins, but slower and more protocol overhead.
Bare metal / superloop
No OS: init peripherals, then one main loop polling flags set by ISRs. Simple, fully deterministic, but everything shares one loop's timing budget.
RTOS
Real-time operating system (FreeRTOS, Zephyr): preemptive priority scheduler, tasks with own stacks, queues/semaphores/mutexes, and bounded response times.
Task / scheduler / preemption
The scheduler always runs the highest-priority ready task, preempting lower-priority work the moment something more urgent becomes ready (e.g. woken by an ISR).
Priority inversion
A low-priority task holding a mutex blocks a high-priority task while medium-priority tasks run. Fix: priority inheritance — the holder temporarily inherits the waiter's priority.
Watchdog timer
A countdown that resets the chip unless firmware "kicks" it periodically — the last-resort recovery from hangs and lockups. Kick from a place that proves the system is actually healthy.
Low-power modes
Sleep/stop/standby states that shut down clocks and power domains. Deeper modes save more current but wake more slowly and may lose RAM/peripheral state; battery devices live asleep and wake on interrupts.
Bootloader
Small first-stage program that can validate and update the main firmware image (over UART/USB/OTA) before jumping to it.
JTAG / SWD
Debug ports used to flash the chip and debug live: halt the core, set hardware breakpoints, inspect registers and memory while the hardware runs.

Core concepts

MCU or application processor?

Choose by timing guarantees, power, and cost: an MCU wakes from sleep in microseconds, runs an ISR with sub-microsecond jitter, and idles at microamps; a Linux-class processor gives you networking stacks, filesystems, and processes but millisecond-scale jitter and seconds of boot. Many products use both — Linux for UI/cloud, an MCU for the motor loop — talking over UART or SPI.

Registers and volatile

Firmware configures hardware by read-modify-write on memory-mapped registers: read, mask, set bits, write back. Two classic pitfalls: forgetting volatile, so the optimizer deletes "redundant" reads of a status register and your busy-wait loop never sees the flag; and RMW races when an ISR touches the same register between your read and write.

ISRs: short, then defer

An ISR blocks everything at or below its priority, so do the minimum: read the data, clear the interrupt flag, set a volatile flag or push to a queue, return. Heavy work happens in the main loop or a woken task. Sharing data with an ISR needs a critical section (briefly disable interrupts) or atomic access — a plain 32-bit counter updated by both sides can tear or lose updates. Forgetting to clear the peripheral's flag re-enters the ISR forever.

DMA: move data without the CPU

At 1 MB/s a byte-per-interrupt UART driver takes a million interrupts a second — DMA turns that into one interrupt per buffer. Ping-pong (double) buffering lets the CPU process buffer A while DMA fills buffer B. Pitfalls: on cached parts the CPU may read stale RAM unless caches are invalidated (or buffers placed in non-cached regions), and the buffer must stay valid for the DMA's lifetime — no stack buffers.

Choosing a serial bus

UART: two wires, no clock, point-to-point, both ends must agree on baud — the default debug/console link. SPI: fastest and simplest protocol, full duplex, but one CS pin per device — pick it for high-rate peripherals like displays and flash. I2C: two wires shared by many addressed devices with ACKs — pick it for low-rate sensor swarms; remember the pull-ups, since the lines are open-drain and can only pull low.

Bare metal vs RTOS, and staying alive

A superloop is right when timing is simple and code is small; an RTOS earns its footprint when independent activities with different deadlines would otherwise turn the loop into a hand-rolled scheduler. RTOS pitfalls: unbounded blocking in high-priority tasks, priority inversion (enable priority inheritance on mutexes), and stack overflows per task. Whatever the architecture, a watchdog guards against the failure you didn't foresee — and the build/flash/debug loop (cross-compiler + linker script → .elf/.hex → flash over SWD → printf on UART or halt in a debugger) is the daily rhythm everything else depends on.

Deeper dives

Atomic register writes vs read-modify-write

GPIOA->ODR |= (1<<5) is three steps — read, modify, write — and an ISR writing the same register in between is silently undone. Hardware designers provide single-write alternatives: set/reset registers (BSRR on STM32), where one store atomically sets or clears chosen bits. Rule: prefer write-only set/clear registers for anything an ISR might also touch; otherwise wrap the RMW in a critical section.

Interrupt plumbing on Cortex-M

The vector table at the start of flash holds the initial stack pointer and one handler address per exception. The NVIC assigns each IRQ a priority (lower number = more urgent), preempts lower-priority handlers (nesting), and tail-chains back-to-back interrupts without re-stacking. Design the priority map deliberately: motor-fault above tick, tick above UART, everything documented — an accidental priority inversion between ISRs is painful to debug.

DMA patterns that actually ship

The workhorse is circular DMA with half- and full-transfer interrupts — ping-pong buffering with zero per-byte CPU cost. On cached cores (M7 and up), clean the cache before DMA reads memory and invalidate before the CPU reads what DMA wrote, or you will chase "corrupted" data that is really stale cache. And DMA is not free design-wise: for a 9600-baud UART, a plain receive interrupt is simpler and just as good.

Bus protocol details that matter

SPI has four modes from CPOL/CPHA (clock idle level, sample edge) — mode mismatch reads shifted garbage; check the peripheral's datasheet timing diagram. I2C reads are two-phase: write the register address, repeated START, then read — and slaves may stretch the clock. UART 8N1 spends 10 bits per byte, so 115200 baud moves at most ~11.5 kB/s — budget your logging accordingly.

RTOS primitives: pick the right one

Queue = data plus wakeup (the default ISR→task channel). Binary semaphore = pure signal, no data. Mutex = ownership with priority inheritance — never lock one in an ISR. Event groups = wait for any/all of several flags. In ISRs, only the *FromISR variants are legal. Size each task's stack with high-water marks, not optimism — stack overflow is the classic "random" RTOS crash.

Watchdog strategy, not just a kick

A watchdog kicked from a timer ISR proves only that the timer works while the application is dead. Better: a supervisor pattern — each task sets its bit in a check-in mask, and one place kicks the dog only when all bits are present, then clears the mask. Windowed watchdogs also catch too-early kicks (runaway loops). On boot, read the reset-cause register and log it — a device that resets in the field should tell you why.

Low power is a duty-cycle design

Average current ≈ Isleep + (tawake/T)·Iawake — the win comes from sleeping longer, not from shaving awake current. Architecture: wake on interrupt (RTC, pin, DMA-complete), do the burst of work fast, return to the deepest mode whose wake latency you can afford. RTOS tickless idle stops the tick interrupt from waking the chip 1000× a second. Let peripherals + DMA gather data while the CPU sleeps.

Finding firmware bottlenecks

The MCU profilers are a GPIO pin and a counter: toggle a pin around the code and measure with a logic analyzer (also shows jitter across runs), or read the DWT cycle counter (CYCCNT) before/after. Overcome bottlenecks by design, not micro-tweaks: move data with DMA instead of loops, generate waveforms with timers/PWM instead of bit-banging, do math in fixed point or CMSIS-DSP, precompute lookup tables, and replace polling loops with interrupts + sleep.

The build → flash → debug loop

Cross-compiler (arm-none-eabi-gcc) turns C into an ELF; the linker script decides what goes in flash vs RAM (vector table first, .data initializers copied to RAM by the startup code); objcopy makes the .bin/.hex; the probe flashes it over SWD. Debug tools in order of intrusiveness: hardware breakpoints (halt the core), RTT/SWO trace (printf without stopping), UART logs, and the GPIO pin. For timing bugs, halting changes behavior — use the non-halting ones.

Coding Interview Practice

Fourteen write-the-code questions repeatedly reported in real interviews (Glassdoor reports for firmware/DSP roles at companies like Cadence, and NVIDIA-style CUDA screens), each with a worked solution and the talking points interviewers probe. Try each one on paper before opening the answer — the follow-up is usually "now optimize it" or "what breaks it?".

Bit manipulation & C fundamentals

1. Count the set bits in a 32-bit integer — then optimize itreported at Cadence

The classic warm-up with a built-in follow-up. Start with the loop everyone knows, then show Kernighan's trick, then know what "even faster" means.

int count_bits(uint32_t x) {          // Kernighan: O(set bits), not O(32)
    int n = 0;
    while (x) {
        x &= x - 1;                   // clears the LOWEST set bit
        n++;
    }
    return n;
}

Why x &= x−1 works: subtracting 1 flips the lowest set bit to 0 and all bits below it to 1; ANDing erases exactly that bit. 0b101100 → 0b101011 → AND → 0b101000.

The "optimize it" follow-up: a 256-entry lookup table processes a byte at a time (4 loads, no branches); real code uses the hardware instruction via __builtin_popcount(x) (POPCNT/VCNT). Saying "I'd use the builtin in production and here's what it compiles to" is the senior answer.

2. Reverse the bits of a 32-bit integer

Loop version first (per-bit, easy to get right under pressure), then the divide-and-conquer version if asked to optimize.

uint32_t reverse_bits(uint32_t x) {   // O(log n): swap halves, then quarters...
    x = (x >> 16) | (x << 16);                            // swap 16-bit halves
    x = ((x & 0xFF00FF00u) >> 8) | ((x & 0x00FF00FFu) << 8);   // bytes
    x = ((x & 0xF0F0F0F0u) >> 4) | ((x & 0x0F0F0F0Fu) << 4);   // nibbles
    x = ((x & 0xCCCCCCCCu) >> 2) | ((x & 0x33333333u) << 2);   // pairs
    x = ((x & 0xAAAAAAAAu) >> 1) | ((x & 0x55555555u) << 1);   // single bits
    return x;
}

Talking points: 5 steps instead of 32 iterations; the masks are self-describing in hex (0xAAAA… = alternating bits). The u suffixes matter — shifting a signed int left into the sign bit is undefined behavior. RBIT does this in one instruction on ARM.

3. Detect endianness at runtime, and byte-swap a 32-bit value

Standard firmware question because every wire protocol (network order = big-endian) meets a little-endian CPU eventually.

int is_little_endian(void) {
    uint32_t x = 1;
    return *(uint8_t *)&x == 1;       // LE stores the LSB at the LOWEST address
}

uint32_t bswap32(uint32_t x) {        // 0x11223344 → 0x44332211
    return  (x >> 24)
         | ((x >>  8) & 0x0000FF00u)
         | ((x <<  8) & 0x00FF0000u)
         |  (x << 24);
}

Talking points: inspecting bytes through a uint8_t* is legal (char types may alias anything); casting to uint32_t* the other way is strict-aliasing UB. Production code uses __builtin_bswap32 / htonl. Mention that endianness bugs hide until the struct crosses the wire.

4. Write SET / CLEAR / TOGGLE / TEST bit macros — and say what's wrong with using them on registers 
#define BIT(n)         (1u << (n))            // 1u, NOT 1: 1 << 31 is UB
#define SET_BIT(r, n)  ((r) |=  BIT(n))
#define CLR_BIT(r, n)  ((r) &= ~BIT(n))
#define TGL_BIT(r, n)  ((r) ^=  BIT(n))
#define TST_BIT(r, n)  (((r) >> (n)) & 1u)

The two traps interviewers listen for: (1) 1 << 31 with a signed int is undefined behavior — hence 1u; every parameter parenthesized. (2) All three modify macros are read-modify-write: on a hardware register shared with an ISR, the ISR's change can be lost between the read and the write — which is why set/reset registers (BSRR) or a critical section are used instead. Connecting the macro to the RMW race is what separates the levels.

Embedded firmware patterns

5. Lock-free ring buffer between a UART ISR and the main loopvery frequently asked

THE embedded interview data structure. Single producer (ISR) — single consumer (main): it can be safe with no locks at all, if you can say why.

#define SIZE 256u                       // power of two — enables masking
static volatile uint8_t  buf[SIZE];
static volatile uint16_t head;          // written ONLY by the ISR
static volatile uint16_t tail;          // written ONLY by main

bool rb_put(uint8_t b) {                // ISR side
    if ((uint16_t)(head - tail) == SIZE) return false;   // full
    buf[head & (SIZE - 1u)] = b;        // write the data FIRST...
    head = (uint16_t)(head + 1);        // ...then publish it
    return true;
}

bool rb_get(uint8_t *b) {               // main-loop side
    if (tail == head) return false;     // empty
    *b = buf[tail & (SIZE - 1u)];
    tail = (uint16_t)(tail + 1);
    return true;
}

Why it's safe with no lock: each index has exactly one writer, indices are free-running (wrap via unsigned arithmetic, so full = head−tail == SIZE and empty = equal, no wasted slot), and data is written before the index that publishes it. Follow-ups: why power-of-two (mask instead of %), what breaks with two producers (RMW on head — now you need a critical section), and on out-of-order CPUs you'd need a memory barrier where the comment says "publish".

6. Debounce a button in firmware (no delays allowed)

Mechanical contacts bounce for ~1–10 ms. Blocking delay_ms(20) is the answer that fails the interview; the shift-register sampler is the one that passes.

// called from a 1 ms timer tick
bool debounce(bool raw) {
    static uint8_t hist = 0;
    static bool stable = false;
    hist = (uint8_t)((hist << 1) | (raw ? 1u : 0u));
    if (hist == 0xFFu) stable = true;      // 8 consecutive 1s → pressed
    else if (hist == 0x00u) stable = false;// 8 consecutive 0s → released
    return stable;                          // anything else: keep last state
}

Talking points: the 8-bit history is a sliding 8 ms window; bounce shows up as mixed bits and simply doesn't change the state. Zero blocking, one byte of RAM, and the same tick can debounce a whole port in parallel with word-wide shifts. Follow-up: hysteresis is inherent — press and release each need 8 clean samples.

7. Implement memmove (memcpy that survives overlap)

Asked to test pointer reasoning: when does forward copying corrupt the data it hasn't read yet?

void *my_memmove(void *dst, const void *src, size_t n) {
    uint8_t *d = (uint8_t *)dst;
    const uint8_t *s = (const uint8_t *)src;
    if (d < s) {
        while (n--) *d++ = *s++;        // dst below src: forward is safe
    } else if (d > s) {
        d += n; s += n;
        while (n--) *--d = *--s;        // dst above src: copy BACKWARD
    }
    return dst;
}

The reasoning: if dst overlaps ahead of src, a forward copy overwrites source bytes before reading them — so copy from the end. memcpy is allowed to assume no overlap (that's why it can be faster); passing overlapping buffers to it is UB that "works" until the libc vectorizes. Follow-up: word-at-a-time copying and alignment handling.

8. Byte-stream packet parser as a state machine

Frame format: 0xAA | len | payload[len] | checksum. Bytes arrive one at a time (UART). Parse robustly — including garbage and truncated frames.

#define MAXPL 64
typedef enum { WAIT_SOF, LEN, DATA, CSUM } state_t;

void parse_byte(uint8_t b) {
    static state_t st = WAIT_SOF;
    static uint8_t len, idx, csum, pl[MAXPL];
    switch (st) {
    case WAIT_SOF:
        if (b == 0xAA) st = LEN;                 // hunt for start of frame
        break;
    case LEN:
        len = b; idx = 0; csum = b;
        st = (len == 0 || len > MAXPL) ? WAIT_SOF   // reject bad length!
           : DATA;
        break;
    case DATA:
        pl[idx++] = b; csum ^= b;
        if (idx == len) st = CSUM;
        break;
    case CSUM:
        if (b == csum) handle_frame(pl, len);    // else: drop silently
        st = WAIT_SOF;                           // resync either way
        break;
    }
}

What they're checking: byte-at-a-time design (no blocking reads), the length bound check (unchecked len is a buffer overflow — the most common bug in submitted answers), checksum accumulated on the fly, and automatic resync after corruption. Follow-up: what if 0xAA can appear inside the payload? (Escape sequences or checksum-validated resync.)

DSP in C

9. Implement an FIR filter in Creported at Cadence

The literal Glassdoor-reported question for DSP roles. The naive version shifts the whole history every sample; the expected version uses a circular buffer.

#define NTAPS 8
static const float h[NTAPS] = { /* coefficients */ };
static float xhist[NTAPS];             // history of inputs
static int   pos;                      // circular write index

float fir(float in) {
    xhist[pos] = in;                   // overwrite the oldest sample
    float acc = 0.0f;
    int j = pos;
    for (int k = 0; k < NTAPS; k++) {  // h[0]·newest ... h[N-1]·oldest
        acc += h[k] * xhist[j];
        j = (j == 0) ? NTAPS - 1 : j - 1;
    }
    pos = (pos + 1 == NTAPS) ? 0 : pos + 1;
    return acc;
}

The follow-up is always "optimize it": power-of-two NTAPS turns the wraps into & (NTAPS−1); symmetric (linear-phase) taps halve the multiplies (h[k] = h[N−1−k] → add the sample pair first); DSP cores do the whole tap in one cycle with dual-MAC and circular addressing hardware; and in fixed point the accumulator needs guard bits (this connects to the Q15 question above).

10. Moving-average filter with O(1) update

Trap version: "average the last N samples" — the naive answer re-sums N values every sample. The running-sum answer is one add, one subtract.

#define N 16u                          // power of two
static uint16_t win[N];
static uint32_t sum;                   // MUST hold N × max sample!
static unsigned idx;

uint16_t moving_avg(uint16_t in) {
    sum -= win[idx];                   // retire the oldest sample
    win[idx] = in;
    sum += in;                         // admit the newest
    idx = (idx + 1) & (N - 1u);
    return (uint16_t)(sum >> 4);       // /16 — shift, not divide
}

Talking points: O(1) vs O(N) per sample; the accumulator sizing (16 × 65535 needs 20 bits — a uint16_t sum overflows silently); integer running sums are exact, so no float drift. Bonus points for naming the frequency-domain view: this is an FIR with rectangular taps — cheap but with poor stopband (first sidelobe −13 dB), and cascading it (CIC) is how decimators fix that.

11. Sine-wave generator with a lookup table (DDS / phase accumulator)

Standard audio/comms firmware question: generate a sine at an arbitrary frequency with no floating point in the sample loop.

static const int16_t sine_lut[256] = { /* one full cycle, Q15 */ };
static uint32_t phase;                 // 32-bit phase accumulator
static uint32_t step;                  // phase increment per sample

void set_freq(float f, float fs) {
    step = (uint32_t)(f / fs * 4294967296.0f);   // f/fs × 2^32 (setup only)
}

int16_t next_sample(void) {
    phase += step;                     // wraps at 2^32 = one full cycle, free
    return sine_lut[phase >> 24];      // top 8 bits index the 256-entry table
}

Why it's elegant: the accumulator overflow IS the phase wrap — no modulo, no drift, and frequency resolution is fs/2³² (~0.01 Hz at 48 kHz). Follow-ups: output quality is limited by the 256-step quantization (interpolate between adjacent entries for ~+30 dB), a quarter-wave table saves 4× ROM using symmetry, and two accumulators 90° apart give sin+cos for I/Q mixing.

CUDA kernels

12. Write a vector-add kernel and its launchthe NVIDIA screener

The canonical CUDA opener. Correctness is assumed; the interview is in the details you mention unprompted.

__global__ void vadd(int n, const float *a, const float *b, float *c) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) c[i] = a[i] + b[i];             // guard the tail
}

// launch: ceiling division covers every element
vadd<<<(n + 255) / 256, 256>>>(n, d_a, d_b, d_c);

// grid-stride version: fixed grid handles ANY n, and re-uses threads
__global__ void vadd_gs(int n, const float *a, const float *b, float *c) {
    for (int i = blockIdx.x * blockDim.x + threadIdx.x;
         i < n;
         i += gridDim.x * blockDim.x)
        c[i] = a[i] + b[i];
}

Say these before being asked: the guard (grid is rounded up), consecutive threads → consecutive addresses (coalesced by construction), the kernel is memory-bound (arithmetic intensity ~1 FLOP per 12 bytes — so tuning math is pointless), and the grid-stride form decouples grid size from data size (also the wave-quantization fix).

13. Parallel reduction (sum) — "how would you realize it?"reported in NVIDIA interviews 
__global__ void reduce_sum(const float *x, float *total, int n) {
    __shared__ float s[256];
    int tid = threadIdx.x;
    int gid = blockIdx.x * blockDim.x + tid;
    s[tid] = (gid < n) ? x[gid] : 0.0f;        // pad the tail with zeros
    __syncthreads();

    for (int stride = blockDim.x / 2; stride > 0; stride >>= 1) {
        if (tid < stride)
            s[tid] += s[tid + stride];
        __syncthreads();                        // OUTSIDE the if — always
    }
    if (tid == 0) atomicAdd(total, s[0]);       // combine block partials
}

The three things they probe: barrier placement (inside the branch = deadlock — the classic filter), how blocks combine (no grid-wide barrier → atomicAdd or a second kernel), and the follow-up ladder: sequential addressing avoids bank conflicts, warp-shuffle (__shfl_down_sync) does the last 32 without shared memory, and float atomics make the result non-bit-reproducible (order varies run to run).

14. Matrix transpose with coalesced reads AND writes

The classic memory-optimization exercise: naive transpose coalesces reads but scatters writes (or vice versa). The fix stages a tile in shared memory.

__global__ void transpose(float *out, const float *in, int w, int h) {
    __shared__ float tile[32][33];             // 33: +1 pad kills bank conflicts
    int x = blockIdx.x * 32 + threadIdx.x;
    int y = blockIdx.y * 32 + threadIdx.y;
    if (x < w && y < h)
        tile[threadIdx.y][threadIdx.x] = in[y * w + x];   // coalesced read
    __syncthreads();

    x = blockIdx.y * 32 + threadIdx.x;         // swap BLOCK coordinates,
    y = blockIdx.x * 32 + threadIdx.y;         // keep thread coords straight
    if (x < h && y < w)
        out[y * h + x] = tile[threadIdx.x][threadIdx.y];  // coalesced write
}

The two ideas: the transpose happens inside shared memory (cheap), so both global accesses stay row-contiguous and coalesced; and the [32][33] padding shifts each row by one bank so the column read tile[threadIdx.x][…] hits 32 different banks instead of one (a 32-way conflict without it). Measured impact of the pad alone is typically ~2× — a favorite "why is this line here?" question.
